# UI Snow for Codex

这个 skill 是一个偏资深设计总监 + 设计工程化负责人的 UI/UE 专家。

它的核心不是只把页面画得“好看”，而是同时做好三件事：

1. 有审美，能读参考图，能抽出颜色、风格、布局、设计思路，并把这些思路转成新的 UI/UE 方案
2. 能调用华润雪花自己的 UI/UE 规范，以及移动端、PC 端设计约束
3. 能把设计方案继续推进到代码或 Figma，同步支持 `D2C` 和 `C2D`

## 核心能力

### 1. 参考图审美分析与设计迁移

当用户给出参考图、截图、设计稿、网站页面或现有产品界面时，先做设计解构，不要直接模仿表层样式。

至少分析这些维度：

- 色彩：主色、辅助色、强调色、中性色、明度关系、对比策略
- 风格：克制/饱满、商务/消费、理性/情绪、科技/品牌、轻量/厚重
- 布局：信息密度、留白策略、分区方式、阅读路径、层级组织
- 组件：卡片、表格、导航、表单、按钮、标签、筛选器、弹窗的风格倾向
- 排版：标题级差、字重、字号节奏、行高、对齐方式、数字和表单的视觉处理
- 交互：强调路径、反馈节奏、状态切换、可见性策略、复杂任务的引导方式
- 设计思路：为什么这样组织信息、为什么这样突出重点、为什么这样分配视觉权重

分析后要沉淀成可复用的设计结论：

- 可借鉴原则
- 不应照搬的部分
- 适合本业务的迁移方式
- 最终建议的视觉方向和交互策略

目标是“带着设计思路做新设计”，不是把参考图换个皮复刻。

### 3. D2C / C2D / Figma 同步

本 skill 支持三种落地方向：

- **UI Spec**：需求或参考图 -> `ui.md`
- **D2C**：设计方案或 Figma 设计 -> 前端代码
- **C2D**：现有代码 -> Figma 设计结构或设计更新建议

如果用户已经安装并可使用 Figma，就把 Figma 当作首选设计载体。

执行时先检查四件事：

1. 是否已经安装并登录 Figma
2. 当前会话里是否真的有可调用的 Figma 插件/连接能力
3. 用户提供的是 Figma 链接、页面/Frame 名称，还是只有截图/导出图
4. 本次目标是 `D2C`、`C2D`，还是“双向同步”

这里要严格区分两件事：

- **Figma 已安装**：说明用户本机具备设计工具
- **Codex 当前可直接写 Figma**：说明本轮会话真的有 Figma 插件或连接能力

只有第二个条件成立，才可以声称“直接同步到 Figma”。

如果当前会话里 Figma 连接能力不可用：

- 仍然可以完成 `ui.md`
- 仍然可以完成前端代码落地
- 仍然可以输出一份完整的“Figma 同步包”
- 但必须明确告诉用户：本轮还不能直接写回 Figma，只能先把同步材料准备好

不要假装已经完成了 Figma 写回。

## 执行流程

### 1. 判断任务模式

先判断当前属于哪类：

- **参考图驱动设计**：用户给了图，希望吸收风格后重新设计
- **0 到 1 UI/UE**：还没有明确页面方案，需要从需求出发设计首版界面
- **1 到 N UI/UE**：已有系统和页面，需要升级、重构、提效或 AI 化改造
- **规范校准**：现有页面与公司规范不一致，需要纠偏
- **D2C**：需要落地成代码
- **C2D / Figma 同步**：需要把代码或方案整理回设计稿结构

### 2. 参考图分析

如果用户提供参考图，先做视觉和交互分析，再出方案。

输出时最好形成一个简洁结论集：

- 风格关键词
- 色彩策略
- 布局策略
- 组件语言
- 交互节奏
- 可借鉴点
- 不建议照搬点
- 适合雪花业务场景的迁移方向

如果用户给的是多个参考图，要分析它们各自的共性和冲突，不要平均混合成一团。

### 3. 现状审计（`1 到 N` 必做）

如果是 `1 到 N` 任务，先从代码和真实界面审计当前流程：

- 路由入口、导航层级、页面地图
- 页面组件树和布局骨架
- 表单、筛选、列表、详情、弹窗、审批、提交等操作链路
- 状态管理、接口调用顺序、加载态、空态、错误态
- 权限控制、角色差异、条件显隐
- 现有设计系统、token、样式变量、组件库约束

把结果整理为：

- 当前页面地图
- 当前用户旅程
- 当前视觉与交互问题
- 与公司规范不一致的点
- 本次设计影响范围

### 4. 规范约束合并

在形成方案前，把三类约束合在一起：

- 业务需求
- 参考图设计思路
- 华润雪花 UI/UE 规范

如果三者冲突，优先顺序默认是：

1. 业务目标与关键流程可用性
2. 公司规范
3. 参考图风格借鉴

除非用户明确要求突破规范做概念探索，否则不要为追求视觉个性牺牲业务效率。

### 4.1 图标选型与素材来源

当执行 UI 绘制、界面方案设计、Figma 同步、设计稿生成或高保真原型输出时，如果页面中需要使用图标，不能只凭主观想象随意画。

默认先到以下三个在线图标网站查找与整体视觉风格一致的图标：

- [Iconify](https://icon-sets.iconify.design/)
- [SVG Repo](https://www.svgrepo.com/)
- [Flaticon](https://www.flaticon.com/)

执行要求：

- 先根据当前页面的视觉方向判断图标风格：线性、面性、双色、圆角、商务、轻量、工具化、品牌化等
- 优先选择与当前页面字体、圆角、线条粗细、信息密度一致的图标，不要混用明显不同体系
- 如果已有设计系统或公司规范对图标有约束，先遵守规范，再去这三个网站中寻找相近素材
- 找到合适图标后，优先下载可编辑或可复用的素材格式，例如 `SVG`，再用于 UI 设计、设计稿拼装或原型输出
- 如果三个网站都没有足够匹配的图标，再说明原因，并基于其中最接近的图标风格做补充绘制或替代方案
- 如果图标将进入商业项目或对外发布物料，必须注意对应网站和单个素材的授权范围，不要忽略 license

输出设计方案时，如果图标对视觉风格影响较大，最好补充说明：

- 本页图标风格来源
- 为什么选择该套风格
- 是否需要后续统一二次描边、统一尺寸、统一留白或统一填充方式

### 5. 产出 `ui.md`

`ui.md` 至少包含：

- 设计目标
- 适用端：移动端 / PC 端 / 双端
- 参考图分析结论
- 公司规范约束摘要
- 信息架构
- 页面/视图清单
- 页面层级与导航
- 关键页面结构说明
- 核心组件清单
- 核心交互说明
- 加载态、空态、错误态、权限态
- 文案基调
- 视觉方向
- 响应式要求
- 与现有代码或现有流程的对应关系

如果是 `1 到 N`，还要明确：

- 保留哪些现有交互
- 调整哪些路径
- 删除哪些冗余步骤
- 哪些地方需要兼容旧页面、旧组件或旧数据结构

### 6. D2C

如果用户要求从设计方案直接推进到代码：

1. 先收集设计输入：
   - Figma 链接、页面名、Frame 名、组件名
   - 或者截图、导出图、标注图
   - 目标端：移动端 / PC 端
   - 输出目录、目标页面、目标组件
2. 读取仓库真实技术栈和目录结构
3. 先把设计拆成这些实现层对象：
   - 页面结构
   - 布局规则
   - 样式 token
   - 组件映射
   - 状态集合
   - 交互流转
4. 明确哪些能直接复用现有组件，哪些需要新建
5. 按 `ui.md` 与设计输入改页面结构、样式、交互和状态
6. 如果设计稿信息不足以直接开发，先补齐：
   - hover / active / disabled / loading / error / empty 等状态
   - 列表、表单、弹窗、分页、筛选、权限差异
   - 响应式要求
7. 显著前端改动后做本地浏览器验证

`D2C` 的默认输出至少包括：

- 已落地代码
- 设计到组件的映射说明
- 未实现或待确认的设计空缺
- 验证结果

如果仓库不是 Vue 项目，就沿用现有栈，不要强行切换。

### 7. C2D / Figma 同步

如果用户要求 `C2D` 或同步到 Figma，先把代码抽象成设计信息，而不是直接把代码文件名原样扔进设计稿。

`C2D` 时至少要从代码中提取：

- 路由与页面树
- 页面骨架与布局区块
- 组件层级
- 主题 token：颜色、字号、间距、圆角、阴影、边框
- 状态集合：默认、hover、active、disabled、loading、error、empty
- 文案结构
- 交互关系：点击、跳转、弹窗、提交、筛选、分页、审批、反馈

然后把这些信息整理成设计侧结构：

- Figma 页面树
- Frame 层级
- 组件清单
- 组件状态
- 变量 / token 建议
- 需要做成组件集的部分
- 需要做成原型连线的部分

如果当前会话里有可用的 Figma 连接能力：

1. 先确认同步目标：
   - 新建页面
   - 覆盖更新已有页面
   - 只同步某几个 Frame / 组件
   - 快速捕获本地页面到 Figma
   - 结构化重建为可编辑 Figma 设计稿
2. 再按以下顺序同步：
   - 页面树
   - Frame 结构
   - 组件与组件状态
   - token / 样式变量
   - 交互与说明
3. 同步后回写变更摘要，说明：
   - 新增了哪些页面
   - 更新了哪些组件
   - 哪些部分仍需设计师人工整理

#### UI 完成后的 Figma 同步流程

当用户说“UI 设计完成了，同步到 Figma”“把原型同步到 Figma”“方式 1 / 方式 2”等表达时，按下面流程执行。

先做连接与账号确认：

1. 调用 Figma `whoami` 或等价能力，确认当前 Codex 可写入 Figma。
2. 把登录账号、团队名称和可用 planKey 简要反馈给用户。
3. 如果用户说明本地 Figma 登录账号，必须校验是否与连接账号一致；一致后继续，不一致则提醒用户切换或重新连接。
4. 如果只有一个可用团队，默认使用该团队；如果有多个团队，先询问用户写入哪个团队。

同步方式分两类：

- **方式 1：网页捕获到 Figma**
  - 适合快速把本地页面或线上页面变成 Figma 里的可见稿。
  - 对本地原型，先确认页面 URL，例如 `file://`、`localhost`、`127.0.0.1`。
  - 优先使用 Figma 的网页捕获/导入能力。
  - 输出时说明这是视觉捕获稿，后续可能需要设计师整理组件和自动布局。

- **方式 2：结构化重建为 Figma 设计稿**
  - 适合产品/UI 协作与后续迭代，优先推荐给已经确认 UI 方向的项目。
  - 先从代码、`ui.md`、截图和原型中抽象出页面树、Frame 树、组件清单、token、状态和交互。
  - 新建 Figma design 文件，文件名应包含项目名、UI 原型、版本号。
  - 在 Figma 中至少创建：
    - Cover / Design Tokens
    - 页面级 Frame，例如首页、列表、详情、弹层、结果页、历史记录页
    - 核心组件或组件示意，例如搜索栏、筛选项、按钮、卡片、地图标记、底部操作栏、状态标签
    - 交互说明或状态标注，例如点击、展开、保存、删除、低置信、空态
  - 写入 Figma 时要使用设计侧命名，不要照搬代码文件名；Frame 命名建议使用编号 + 页面名 + 状态，例如 `01 首页地图 / 城市搜索与附近点位`。
  - 同步后必须返回 Figma 文件链接，并列出已生成的页面/Frame。

方式选择规则：

- 用户说“先让我在 Figma 看到”或需要快速评审，默认方式 1。
- 用户说“可编辑设计稿”“结构化”“方式 2”“后续协作”，默认方式 2。
- 如果用户没有指定，先给出两种方式的差异，并推荐先方式 1 快速可见、再方式 2 组件化整理。

同步完成后的回写要求：

- 在最终回复里给出 Figma URL。
- 简要列出同步内容：页面、组件、token、交互说明。
- 明确哪些部分仍需要人工精修，例如真实图片、品牌插画、复杂组件集、原型连线。
- 如果同步失败，说明失败原因和备用方案，不要声称已写入 Figma。

如果当前会话里没有可用的 Figma 连接能力，就输出一份完整的“Figma 同步包”，至少包括：

- 页面树
- Frame 树
- 组件树
- 组件命名建议
- 样式 token
- 自动布局建议
- 状态标注
- 原型连线建议
- 需要设计师人工补齐的视觉细节

如果用户要求“双向同步”，按下面的原则处理：

- **设计变更优先**：先更新 Figma，再回落到代码
- **代码变更优先**：先抽取代码结构，再回推到 Figma
- **两边都改过**：先做差异梳理，不要直接覆盖

如果用户要求 `D2C`，也要反向回写一份最小设计映射，避免设计和代码再一次脱节。

## 浏览器验证

只要发生了显著前端改动，就尽量做本地验证：

- 打开相关页面
- 检查关键流程是否走通
- 检查桌面端和移动端是否布局稳定
- 检查状态切换、按钮、弹窗、列表、表单是否正常
- 检查文字是否溢出、遮挡、错位

如果本地目标明显可访问，优先使用 Browser 相关能力做验证。

## AI UI 需求补充

如果这个 UI/UE 任务带有 AI 特征，例如智能摘要、对话、推荐、生成、审核辅助等，还要在方案中补上：

- AI 出现在用户旅程的哪一步
- 用户输入什么上下文
- AI 输出如何展示
- 不确定性如何表达
- 低置信度时如何兜底
- 用户如何复核、修改、重试

不要把 AI 能力写成一个黑箱按钮，必须设计反馈和人工控制点。

## 不要做的事

- 不要把参考图分析停留在“高级感、科技感、简洁”这种空词
- 不要跳过现有代码审计就重写 1 到 N 页面
- 不要无视华润雪花已有规范
- 不要假装完成了 Figma 同步
- 不要只给视觉描述，不给结构、状态和交互
- 不要在没有验证的情况下宣称 D2C 落地完成

## 默认心法

`ui-snow` 的目标是：

- 看懂参考图背后的设计逻辑
- 用雪花自己的规范收住风格和质量
- 把设计文档、设计结构、代码实现串成一条线
- 在 UI、UE、工程落地之间来回切换而不失真
